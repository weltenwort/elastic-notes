Mitigation for field limit being reached

  • drop and ignore field
  • increase field limit setting for index
  • change field type to flattened to unify multiple fields
  • partition and re-route into separate data streams with smaller sets of fields each

Mitigation for ignore_above limit being reached

  • drop field
  • change field type to text
  • truncate or reduce value length another way
  • increase ignore_above limit setting for field

Mitigation for malformed values being ignored

  • drop field
  • transform value, depends on cause
  • change field format setting in mapping
  • change field type in mapping
  • if conflicting field types for different types of log entries, re-route into separate data streams with non-conflicting field types

Diagnosis of ignored fields

  • need heuristic until ES records the reason
  • example heuristic:
    • field limit reached and field not in mapping -> caused by field limit
    • field has ignored_above and value is longer -> caused by ignored_above
    • otherwise caused by ignore_malformed

Diagnosis of empty fields

  • could detect this by comparing to lists of required or recommended fields

Diagnosis of unsuitable values

  • could detect this by comparing the values to a set of well-known suitable values
  • useful for categorical fields:
    • log.level

Diagnosis of mapping parsing failure

  • error.message field contains details like [1:15] failed to parse field [@timestamp] of type [date] in document with id 'yVN6NpAByace3B8h-c5t'. Preview of field's value: 'abc'
  • could be parsed into
    • field
    • type
    • _id
    • the value that caused the failure

Diagnosis of pipeline processor error

  • error.pipeline contains the pipeline containing the failing processor, such as "logs@custom"
  • error.pipeline_trace contains the call stack of involved pipelines, such as ["logs@default-pipeline", "logs@custom"]
  • error.processor_type contains the type of the processor, such as json
  • error.message contains details, but the format is diverse
  • the combination of error.processor_type and error.type is not sufficient to disambiguate the error.message format

Diagnosis of mismatching field types

  • can only occur if not the complete ECS mapping has been applied, otherwise it would cause a mapping error or be silently coerced
  • could detect this by comparing to a simulated index with full ECS mappings
    • dynamic mappings mean we have to use a source document to ensure the relevant fields are materialized
    • exhaustive iteration over field pairs
